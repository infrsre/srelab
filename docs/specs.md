## Why Specs Matter

When you ask an AI agent to "build a PagerDuty bot", it makes dozens of hidden decisions: which file to create first, whether to write tests, what format the API calls use, how secrets are stored. Those decisions may be wrong for your team — and you only find out when reviewing a wall of code at the end.

**Specs flip this dynamic.** You write your constraints upfront, and Kiro generates work that matches them — one reviewable task at a time.

    <ul>

      <li>Constraints in Requirements become constraints in the code</li>
      <li>Every task is a single, focused diff you can accept or reject</li>
      <li>Tasks run in dependency order — no wasted work</li>
      <li>Re-running a rejected task picks up exactly where you left off</li>

## Spec Structure

Every spec has three sections. Together, they give Kiro everything it needs to build the right thing the right way.

<div className="spec-anatomy">
  <div className="spec-part spec-part--req">
    <div className="spec-part__num">01 — REQUIREMENTS</div>
    <strong>Requirements</strong>
    <p>What this must accomplish. Written as user stories or acceptance criteria. This is the <em>why</em> — and the constraints that cannot be violated.</p>
  </div>
  <div className="spec-part spec-part--design">
    <div className="spec-part__num">02 — DESIGN</div>
    <strong>Design</strong>
    <p>How it should be built. File layout, technology choices, architecture decisions. This is the <em>how</em> — the shape of the solution.</p>
  </div>
  <div className="spec-part spec-part--tasks">
    <div className="spec-part__num">03 — TASKS</div>
    <strong>Tasks</strong>
    <p>The ordered list of things Kiro will execute. One file or change per task. This is the <em>what</em> — generated by Kiro, reviewed by you.</p>
  </div>
</div>

Kiro reads **Requirements** and **Design** to understand intent, then generates **Tasks** automatically. You review and approve each task's diff before it is written to disk.

## Demo: Incident Alert Bot

This example walks through a complete spec for a real SRE project — a Lambda function that receives PagerDuty webhooks and posts incident alerts to Microsoft Teams. You will see what each section looks like in Kiro, what gets generated, and why each part of the spec matters.

**Spec file:** `.kiro/specs/incident-alert-bot.md`

### Step 1 — Open the Spec Editor

Use the Kiro command palette (`Cmd/Ctrl + Shift + P`) and select **Kiro: New Spec**. Give it the name `incident-alert-bot`. The editor opens with three empty sections ready to fill in:

![Kiro opens the spec editor with Requirements, Design, and Tasks sections — the starting point for any new project](/img/tutorial/specs-start.gif)

---

### Step 2 — Write Requirements

Requirements describe what the system must do and what constraints it must respect. **Constraints are as important as features** — writing "secrets must come from SSM Parameter Store" tells Kiro to use `boto3` SSM calls rather than `os.environ`, a meaningful architectural difference it would otherwise guess at.

![The Requirements section in the Kiro spec editor — user stories and constraints written in plain English](/img/tutorial/specs-requirements.png)

```markdown
## Requirements

- As an on-call SRE, I need PagerDuty alerts automatically posted to Microsoft Teams
  so that the whole team is notified of incidents without anyone manually copy-pasting
  from PagerDuty.

- When a PagerDuty "trigger" event fires, post a MessageCard to the #incidents
  Teams channel containing:
    - Severity badge (P1/P2/P3/P4) with color coding via themeColor
    - Incident title and affected service name
    - On-call engineer's name
    - Runbook URL looked up by service name
    - Link back to the PagerDuty incident

- When a PagerDuty "resolve" event fires, post a "✅ Resolved" follow-up to Teams
  referencing the original incident title.

- The bot must be a Python AWS Lambda function triggered by API Gateway.
- Secrets (Teams webhook URL, PagerDuty signing secret) must come from
  AWS SSM Parameter Store — never hardcoded or read from environment variables directly.
- All errors must be logged to CloudWatch and must not crash the Lambda
  (return HTTP 200 to PagerDuty even on soft errors so it stops retrying).
```

---

### Step 3 — Write Design

Design describes how to build it — file layout, technology choices, and constraints for Kiro's code generation. The more specific you are here, the less Kiro has to guess.

![The Design section in the Kiro spec editor — file layout, technology stack, and architecture decisions](/img/tutorial/specs-design.png)

```markdown
## Design

### Project layout
- src/handler.py    — Lambda entry: validates PagerDuty signature, routes trigger/resolve
- src/pagerduty.py  — PagerDutyEvent dataclass + HMAC signature validation
- src/teams.py      — TeamsClient class, build_card() using Teams MessageCard format
- src/runbooks.py   — load_runbooks() reads runbooks.yaml, returns URL by service name
- tests/            — pytest, one file per module, fixtures in tests/fixtures/
- terraform/        — Lambda + API Gateway HTTP API (v2), SSM via data sources

### Python details
- Python 3.12, no web framework (plain Lambda handler)
- Use dataclasses for PagerDuty event models
- Teams MessageCard format: @type, themeColor, sections with facts, potentialAction
- Severity color map: P1=#FF0000, P2=#FF8C00, P3=#FFC200, P4=#36A64F
- Use urllib.request (stdlib) for HTTP — no requests dependency

### Terraform details
- AWS Lambda (zip from src/ + requirements.txt)
- API Gateway HTTP API with POST /webhook → Lambda integration
- IAM role: AWSLambdaBasicExecutionRole + ssm:GetParameter on /incident-bot/*
- No VPC — public Lambda is correct for a webhook receiver

### GitHub Actions
- On push to main: pytest → zip → terraform apply
- Terraform state in S3 backend (bucket from secret TF_STATE_BUCKET)
```

### Step 4 — Generate and Review Tasks

Click **Generate Tasks** in the Kiro sidebar. Kiro reads both sections and produces an ordered task list in about 10 seconds:

![Kiro generates the task list from your Requirements and Design — 14 discrete, reviewable units of work](/img/tutorial/specs-tasks.png)

```markdown
## Tasks

- [ ] 1.  Create requirements.txt (boto3, pytest)
- [ ] 2.  Create src/pagerduty.py — PagerDutyEvent dataclass + validate_signature()
- [ ] 3.  Create src/runbooks.py — load_runbooks() from runbooks.yaml
- [ ] 4.  Create runbooks.yaml with example service entries
- [ ] 5.  Create src/teams.py — TeamsClient, build_card() with MessageCard format
- [ ] 6.  Create src/handler.py — Lambda entry point, route trigger vs resolve
- [ ] 7.  Create tests/fixtures/pagerduty_trigger.json — sample webhook payload
- [ ] 8.  Create tests/test_handler.py — test trigger and resolve paths
- [ ] 9.  Create tests/test_teams.py — test MessageCard structure and severity colors
- [ ] 10. Create terraform/variables.tf
- [ ] 11. Create terraform/iam.tf — Lambda role with SSM permissions
- [ ] 12. Create terraform/main.tf — Lambda + API Gateway HTTP API
- [ ] 13. Create terraform/outputs.tf — webhook_url output
- [ ] 14. Create .github/workflows/deploy.yml — test + terraform apply pipeline
```

Notice the order: `pagerduty.py` is Task 2 because `handler.py` (Task 6) imports it. Kiro resolves these dependencies from your Design section — you don't have to specify order manually.

**Scan the list before running.** Add tasks if something is missing (a `Makefile`, a `README.md`, a Docker image). Remove tasks that don't apply. When the list reflects exactly what you want, click **Run Spec**.

---

### Step 5 — Watch and Review

Kiro executes each task and shows you the diff before writing anything to disk:

![Kiro agent executing tasks in real time — each task appears as a reviewable diff before being written to disk](/img/tutorial/spec-task.gif)

For each diff you can:
- **Accept** — write the file to disk and continue
- **Edit then Accept** — modify the diff inline before accepting (use this when Kiro misses a detail, like a missing `import`)
- **Reject** — skip this task; add a note explaining why so the next run corrects it

Nothing is written without your approval. The spec remains a living document — you can edit Requirements or Design mid-run and re-generate specific tasks.

## Writing Effective Specs

**Be specific in Requirements.** Vague requirements produce vague output.
- Weak: `"Create a monitoring script"`
- Strong: `"Create a Python script that checks ALB 5xx rate via CloudWatch every 5 minutes and pages PagerDuty if the rate exceeds 1% over a 15-minute window"`

**Put constraints in Requirements, not just features.** Lines like "secrets must come from SSM" and "return HTTP 200 even on soft errors" directly shape the code Kiro writes.

**Reference existing patterns in Design.** Tell Kiro what already exists in the repo.
- `"Follow the module structure in modules/rds/"`
- `"Use the logging setup from lib/logger.py"`
- `"Match the test fixtures pattern in tests/test_alerts.py"`

**Keep tasks atomic.** One file or one meaningful change per task. Kiro handles each task independently — small tasks are easier to review, easier to reject, and easier to retry with corrections.

**Iterate.** You can edit a spec mid-run, re-generate tasks, and continue. Specs are living documents — they don't need to be perfect before you start.
